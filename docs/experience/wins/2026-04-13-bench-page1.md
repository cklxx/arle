## Delta vs. `2026-04-09-bench-l4-qwen3-4b`
- C=1 at 128/128: 30.1 t/s → **30.1 t/s** (no change — within noise)
- ITL p50 range: 33.1–36.4 ms → **33.0–36.6 ms** (flat)
- Peak @ C=4: 112.0 → **111.4** (−0.5 %, noise)
- C=2 at 512/256: 57.3 → 57.0, C=4 at 512/256: 112.0 → 111.4

Conclusion: zero throughput regression from 2026-04-09 to 2026-04-13
main on L4 at C≤4.

## Known issue (pre-existing, `2026-04-09-bench-l4-qwen3-4b.md` §Known Issue)
C≥8 all return 0 tok/s / 0ms TTFT with `errors=0` because the server's
CUDA context dies once a batched-decode step hits
`CUDA_ERROR_ILLEGAL_ADDRESS` during batched sampling
(`infer::scheduler::cuda::decode:256 Batched sampling failed`), after
which every subsequent request's prefix-cache KV migration also fails
with the same sticky error. Requests return with 0 tokens, which the
client counts as empty but non-errored streams.

First crash in run #1 at `2026-04-13T12:15:25.638145+00:00`, i.e. after
~780 s of warm traffic. **Not caused by A/B/C/D/E/F/G/H/I**: the same
failure reproduces on the pre-A commit `37a8a82` (see
`docs/experience/errors/2026-04-13-batched-decode-high-concurrency.md`).

The valid data for this baseline is the C=1–4 block. For P0 `page16`
comparison, use those rows.

## Rule
- L4 24GB baseline for `page_size=1` (pre-P0): 30 tok/s C=1, ~111 tok/s C=4.
- Pre-P0 ITL floor: ~33 ms (decode GEMV bound, unchanged since 2026-04-09).
- Any P0 `page16` delta measurement must compare against THIS snapshot
  under identical `--num-slots 4` and default `cuda_graph=true` settings.
- C≥8 data is unusable on this server until the pre-existing batched
  sampling regression is fixed. Baselines at C≥8 are intentionally
  left as 0 rows — not a P0 regression.
