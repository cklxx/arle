# M5.3a + M5.3b-add — device-resident tensor landed (autograd training path)

Status: **architecture win, perf bench `pending-remote`** — see §Results.

## Goal

Transition `crates/autograd/` from *upload-op-download-per-call* to
*device-resident tensors with lazy eval*, so Metal/CUDA matmul and
elementwise chains can compose a single MLX/CUDA graph per training step
instead of one per op. This is the structural unlock behind the 2026-04-18
`bench-train-multi-turn.md` findings (Metal TinyLM 1.9× slower than CPU
because every `matmul → eval → readback` built a 1-op graph — MLX's
degenerate path).

## Hypothesis

1. The architecture transition is numerically neutral — parity tests
   on the ≤1e-3 tolerance continue to pass on both Metal and CPU for
   matmul and add.
2. Wall-clock changes vs the 2026-04-18 baseline will be **small and
   mixed** until more ops are ported. Today only `matmul` and `add` take
   the lazy path; the rest of the forward still forces `ensure_host` each
   call, which means intermediate readback still happens. The *real* perf
   win lands once softmax/rmsnorm/gelu/mean/sum/embedding are ported (C3–C5
   of the CUDA plan and their Metal twins).
3. The crossover point from the 2026-04-18 snapshot
   (Metal wins somewhere between d_model=64 and d_model=256) should not
   regress. Worst case is break-even at d_model=256 because the matmul
   kernel itself is unchanged — only its surrounding call shape changed.

- CPU parity tests: 14 green (unchanged).
- Metal parity tests: 7 green. Add this session: `metal_backend_add_matches_cpu_2d`,
  `metal_backend_add_matches_cpu_3d`, both ≤1e-3 vs CPU reference.
- CUDA typecheck gate (`--features cuda,no-cuda`) green; kernel execution
  stays `PENDING REMOTE CUDA VERIFICATION` — tracked in
  [`docs/plans/m5.3b-cuda-autograd-kernels.md`](../../plans/m5.3b-cuda-autograd-kernels.md)
  phases C1 → C6.
- Clippy `-D warnings` clean on both default + `cuda,no-cuda` feature sets.

**Perf (`pending-remote` / deferred):**

No training wall-clock rerun in this entry. Rationale:

1. Only 2 of ~10 forward ops now take the lazy path. The remaining ops
   (softmax, rmsnorm, gelu, mean, sum, embedding) still force `ensure_host`
   on their inputs, so the per-step graph count has dropped from ~10-per-op
   to ~8-per-op — not the order-of-magnitude reduction that moves the
   `bench-train-multi-turn.md` numbers.
2. Re-running the 2026-04-18 sweep now would produce noise — it should
   fire once C2-C4 have landed on Metal and the full forward composes
   into a single MLX graph.
3. CUDA wall-clock is blocked on remote verification of the C1 NVRTC path
   (in-flight Codex agent; see plan).

## Problems

- **mlx-sys invariant #5 ("only one crate links mlx-sys")** is already
  violated by `crates/autograd/` since M5.3a.1. Pre-existing; not fixed
  here. Documented in working memory; revisit when the training runtime
  moves toward a higher-level Metal facade that can re-share the bridge.
- **`#[allow(irrefutable_let_patterns)]`** on `CpuBackend::add` — without
  the `metal`/`cuda` features, `DeviceHandle` is a single-variant enum so
  the `let DeviceHandle::Cpu(...) = handle` destructure is irrefutable.
  The `#[allow]` is a local cost that lets the CPU-only build stay warning-clean;
  it falls away once another variant is active.

## Learnings

- Device-resident tensor transitions are best shipped op-by-op with parity
  tests per op — not as a single-commit rewrite. Each op's `Backend::<op>`
  trait method is a natural atomic commit unit (done: `matmul`, `add`;
  pending: `mul`, `mul_scalar`, `gelu`, `exp`, `softmax`, `log_softmax`,
  `sum`, `mean`, `sum_broadcast`, `rmsnorm`, `embedding` + backward).
- MLX `logsumexp` / `softmax` / `mean` are all already exposed at the C++
  layer; the FFI-extension commit `e82d9d6` unlocks them with zero kernel
  writing on the Metal side. CUDA equivalents are in the NVRTC plan.
- Splitting CUDA execution off to a remote verification loop is the right
  move for this project — the Mac `cuda,no-cuda` typecheck gate catches
  >90% of build regressions, and the remaining verification (kernel
  semantics under NVRTC) doesn't benefit from iteration speed anyway.

## Rule

Every new op ported to the `Backend` trait lands with (1) CPU impl, (2)
Metal impl + parity test, (3) CUDA `todo!()` stub under `no-cuda` with a
`PENDING REMOTE CUDA VERIFICATION` marker, (4) Mac typecheck gate green.
Training-speed bench entry is written **once** at architectural milestones
(e.g. "full Metal forward composes into one eval"), not per-op.
