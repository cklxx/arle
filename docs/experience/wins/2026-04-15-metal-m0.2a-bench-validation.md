# 2026-04-15 · Metal M0.2a bench validation

## Context

The 2026-04-15 `M0.2a` tranche added resumable Metal request state for Qwen3
and Qwen3.5, but it did **not** yet rewire `metal_serve` onto a live scheduler.

That made the benchmark goal very specific:

1. prove the structural request-state work did not regress the current direct
   Metal path
2. prove the serving shape is still the old serial shape, so this tranche is
   not misreported as scheduler progress

Environment:

- Machine: Apple M4 Pro
- OS: macOS 26.3.1(a) (`Darwin 25.3.0`)
- Build:
  `cargo build --release -p infer --no-default-features --features metal,no-cuda --bin metal_bench --bin metal_serve`
- Artifacts:
  - `/tmp/metal-bench-2026-04-15-m0.2a/qwen35-128-128.json`
  - `/tmp/metal-bench-2026-04-15-m0.2a/metal-serve-quick.json`

## What Worked

The direct path stayed effectively flat relative to the earlier 2026-04-15
API/DX validation snapshot:

| Metric | 2026-04-15 API/DX reference | 2026-04-15 M0.2a validation | Delta |
| --- | ---: | ---: | ---: |
| prompt TPS | `732.9` | `736.1` | `+0.4%` |
| generation TPS | `80.2` | `82.2` | `+2.5%` |
| repo E2E TPS | `72.30` | `73.95` | `+2.3%` |
| TTFT | `174.7 ms` | `173.9 ms` | `-0.5%` |

The quick HTTP sweep also stayed in the same regime:

- `128 / 128, C=1`: `70.0 tok/s`, `TTFT p50 204 ms`
- `512 / 256, C=1`: `66.6 tok/s`, `TTFT p50 598 ms`
- `512 / 256, C=4`: `65.8 tok/s`, `TTFT p50 7994 ms`

Interpretation:

- no meaningful regression from the request-state slice
- `C=4` throughput is still basically flat versus `C=1`
- TTFT at `C=4` is still dominated by queueing, so `M0.2b` remains the real
  serving milestone

## Rule

When a Metal tranche is intentionally structural, the benchmark exit should be
**non-regression plus honest shape preservation**:

- direct path must stay healthy
- serving must not get worse
- if the concurrency curve is still serial, say so explicitly instead of
  claiming a scheduler win
