# M3.9 — Mixed Policy Gate + Full Mixed Prefill Budget

## Context

Phase 1A v3 proved the async readback ring was correct substrate, but the
longctx 4k/c=4 owner-run fired kill criteria: default Mixed scheduling regressed
TTFT from `3403 ms` to `7905 ms` and output throughput from `122.5` to
`69.9 tok/s`.

Source survey confirmed this was not a kernel issue. Mixed scheduling capped
the whole mixed prefill token budget at `long_prefill_token_threshold`
(`4096`), while split/pure prefill could use the full `max_prefill_tokens`
budget (`16384`). That forced c=4 long prompts into one prefill row per mixed
tick.

## What Worked

- Added `--scheduler-mixed-policy split|mixed`.
- Defaulted production serving to `split`, restoring the pre-Phase-1A-v3
  production route while keeping the multi-slot readback ring available.
- Kept `mixed` as an explicit opt-in experiment path.
- Fixed `mixed_prefill_token_budget()` to return the full
  `max_prefill_tokens` whole-step budget. The per-request long-prefill cap is
  still applied when prefill candidates are created.
- Gated mixed workspace reserve behind the same policy, so default `split`
  serving does not shrink the KV pool for a disabled mixed path.

## Bench Status

Status: `pending-owner-run`.

Expected validation:

- Default run: longctx 4k/c=4 returns to the F4-Small range, with
  `plan=split > 0` and `plan=mixed=0`.
- Opt-in run with `--scheduler-mixed-policy mixed`: mixed can pack up to the
  full prefill step budget instead of one 4096-token row per tick.

## Rule

When decode-active prefill has both a per-request row cap and a whole-step cap,
do not reuse the per-request cap as the whole-step budget. That silently turns
cross-request packing off.
