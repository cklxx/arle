# 2026-04-15 · Metal M0.2b live runtime local validation

## Context

`M0.2a` landed resumable Metal request state, but `metal_serve` still ran
through `BackendRuntimeHandle`, so HTTP concurrency stayed effectively serial.

The `M0.2b` local tranche rewired standard `metal_serve` onto a live Metal
scheduler runtime:

- `IncomingRequest` now enters a Metal scheduler loop
- scheduler decisions drive request-state prefill / decode / cancellation
- standard Metal serving no longer imports the serial runtime
- Metal DFlash stays on the old serial fallback because speculative decode is
  not yet wired into the scheduler runtime

During validation, the first implementation regressed TTFT badly because Qwen3.5
request-state prefill was stepping prompt tokens one-by-one through
`cpp_model.step()`. The final local batch fixed that by adding a C++ chunk
prefill bridge (`qwen35_compiled_prefill`) and routing Qwen3.5 prompt chunks
through it.

Environment:

- Machine: Apple M4 Pro
- OS: macOS 26.3.1(a) (`Darwin 25.3.0`)
- Build:
  `cargo build --release -p infer --no-default-features --features metal,no-cuda --bin metal_serve --bin metal_bench`
- Artifacts:
  - `/tmp/metal-bench-2026-04-15-m0.2b/qwen35-128-128.json`
  - `/tmp/metal-bench-2026-04-15-m0.2b/metal-serve-quick.json`
  - `/tmp/metal-bench-2026-04-15-m0.2b-sanity.json`

## What Worked

The serving architecture changed in the intended direction:

- standard `metal_serve` now runs on a live Metal scheduler runtime
- warmup and chat-completions smoke requests still complete normally
- request cancellation / disconnect cleanup is owned by the new runtime rather
  than the old whole-request serial wrapper

The benchmark story is mixed but honest:

- direct `metal_bench` stayed healthy
- single-request HTTP performance stayed near the previous serial baseline
- `C=4` TTFT improved sharply because queue-level FIFO waiting is no longer the
  dominant shape
- aggregate throughput at `C=4` still did **not** beat the old serial runtime,
  which means `M0.2` is still not fully shipped

## Rule

When a serving milestone rewires runtime architecture, the benchmark exit must
separate **latency-shape progress** from **throughput-shape progress**:

- if TTFT collapses but aggregate throughput still trails the old baseline,
  mark the runtime rewiring as real progress but do **not** call the serving
  milestone shipped
- for Qwen3.5 Metal specifically, any resumable request-state path that uses
  `cpp_model.step()` token-by-token for prompt prefill will massively distort
  TTFT; prompt chunks need a batch-prefill bridge before scheduler numbers are
  meaningful
