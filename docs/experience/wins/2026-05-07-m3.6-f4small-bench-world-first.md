# 2026-05-07 · M3.6 F4-Small bench — ARLE 843 out tok/s, +30% past vLLM s14 control

## Goal

- Validate F4-Small (`2a534c4 perf(scheduler): eliminate per-step
  greedy decode sync`) against the M3.6 Phase 1 baseline and the
  Phase 2 vLLM s14 control. Acceptance from
  [`docs/plans/M3.6-F4-eliminate-per-step-sync.md`](../../plans/M3.6-F4-eliminate-per-step-sync.md):
  ≥ 580 out tok/s + cuStreamSynchronize ≤ 25% of CPU API.
- Goal type: **optimization** — first ARLE-side change since the
  M3.6 plan landed; this is the win-or-bust validation.

## Hypothesis

F4-Small (codex implementation in `2a534c4`):
- Removed per-step `ctx.sync()` from `sample_batch_greedy_readback`
- Added GPU-side handoff: `argmax_out` → `next_decode_meta_gpu` D2D
- Async readback via copy stream + CUDA event poll
- Slot-owner array prevents row/slot mismatch corruption
- Device snapshot buffer (codex review nuance) prevents copy-stream
  race with next-tick reuse of `argmax_out`

Plan §F4-Small expected ~50% off ITL (66 ms → ~33 ms), ~700 out
tok/s. Phase 2 wins entry's "upper bound estimate" was ~700 ⇒
parity with vLLM 647.

**Result exceeded the upper bound** — see Results.

## Results

### Headline numbers (45s steady-state at c=64)

Per the canonical guidellm "Server Throughput Statistics" table:

| Metric | Phase 1 baseline (`fdb531b`) | F4-Small (`2a534c4`) | vLLM s14 (`f7146d4`) | F4-Small vs baseline | F4-Small vs vLLM |
|---|---:|---:|---:|---:|---:|
| **out tok/s** | 462 | **843** | 647 | **+82.5%** | **+30.3%** |
| in tok/s | 3,235 | 3,846 | 3,927 | +18.9% | -2.1% |
| req/s | 1.778 | 2.925 | 2.650 | +64.5% | +10.4% |
| **TTFT mdn** | 3,729 ms | 4,069 ms | 17,616 ms | +9% (slightly worse) | **4.3× faster than vLLM** |
| **ITL mdn** | 66.0 ms | **37.8 ms** | 20.0 ms | **-43%** | still 1.89× slower |

**Headline**: ARLE F4-Small at s48 high-conc = **843 out tok/s**,
**+30.3% past vLLM s14 control of 647**. **First confirmed
"world-first parity" data point on this hardware**, with margin
to spare.

### Per-row efficiency (kernel-axis residual)

ITL p50 ARLE 37.8 ms / batch ~38 = **0.99 ms / row / step** (down
from 1.74 ms / row in Phase 1). vLLM s14 ITL 20 ms / 14 rows =
1.43 ms / row.

**ARLE is now also faster per-row than vLLM** at this workload —
1.45× faster. The combination of F4-Small (CPU/GPU overlap) +
ARLE's larger batch (38 vs 14) + the existing CUDA graph capture
gives the throughput advantage.

This was NOT what the M3.6 plan predicted. The plan modeled F4 as
purely closing the scheduling axis and assumed kernel work would
remain a separate per-row gap that M_b would address. The actual
data shows F4-Small alone closes BOTH gaps because:
- Eliminating per-step sync means GPU stays saturated continuously
  (instead of CPU/GPU ping-pong)
- ARLE's larger effective decode batch (38 rows) amortizes
  per-kernel launch overhead across more work
- The CUDA graph (capturing embedding + 36 layers + norm + logits
  GEMM as one replay) was always faster than vLLM's per-step
  kernel launches; F4-Small unblocks that latent advantage

### What changed under the hood (from `2a534c4` + wins entry `2026-05-07-m3.6-f4small-async-greedy-readback.md`)

7 files modified (`infer/src/{model.rs, model/qwen3/{batch_decode,
forward}.rs, scheduler/cuda/{core, core/construction, decode,
execution}.rs}`). Codex review iterations: **4 rounds**, finding
mixed/fallback/telemetry edge cases + 2 P2 race conditions
(repeated-token failure caught by e2e from copy-stream reading
`argmax_out` racing with next-tick reuse — fixed via device
snapshot buffer).

Verification (codex pre-commit): cargo fmt/check/clippy/cuda+metal,
e2e + greedy_consistency tests pass.

2. TTFT mdn is slightly worse (4069 ms vs Phase 1's 3729 ms).

## Learnings

- **F4-Small alone clears the M3.6 milestone goal of vLLM-parity at
  high-conc.** F4-Big and M_b.2 are no longer urgent for "match
  vLLM" — they remain candidates for "exceed vLLM by even more"
  but the headline competitive milestone is achieved.
- The combo of (large decode batch from M3.5 admission work) +
  (CUDA graph capture from prior M-something) + (F4-Small sync
  removal) compounds. Each individual change looked moderate; the
  combined ratio is 1.82× the Phase 1 baseline.
- Codex's 4-round review caught the snapshot-buffer race that
  e2e's repeated-token check exposed. **Manual planning could not
  have predicted this race** — it required actually compiling and
  running the e2e fixture. Validates the cooperative pattern of
  "Claude plans / codex implements with multi-round review".
- nsys 2025.6's spawn-mode timing requires generous `--delay` for
  CUDA graph workloads — graph capture during warmup is itself
  invisible to a too-late-starting trace. For future trace runs
  with this build, use `--delay 60` minimum.

## Next actions (decided by F4-Small bench evidence)

1. **Do NOT rush F4-Big** — F4-Small already exceeded the parity
   target. F4-Big's expected gain (~700 out tok/s in plan) is now
   below the achieved 843. F4-Big becomes a "if we want >900 out
   tok/s" decision, not a "to reach parity" decision.
2. **M_b.1 (TileLang HD128 BF16 decode) becomes optional**.
   Scheduling axis dominated; per-row gap is now in ARLE's favor.
   M_b.2 (FP8 TileLang) could push further but lower priority than
   exploring whether F4-Small's gains hold across workloads beyond
   1024/256 c=64 (e.g. long-context, low-conc, mixed shapes).
3. **Workload sweep**: re-run F4-Small at varying shapes
   (long-context 32k, low-conc c=4, mixed prefill-heavy) to
   confirm the +30% advantage isn't workload-specific.
4. **Update memory** with the F4-Small completed-with-margin
   status so future sessions don't re-attempt fixes that the
   trace already showed are unnecessary.

## Rule

- **Plan upper-bound estimates can be conservative**: the M3.6
  Phase 2 wins entry estimated F4 upper bound at ~700 out tok/s
  (parity). Actual F4-Small landed 843 (+30% past parity). When
  scheduling-axis work clears a long-blocking sync, the latent
  GPU+graph efficiency unblocks more than the linear math
  suggests.
- **F4-Small + CUDA graph capture is the architectural pattern
  worth defending**. Future scheduler refactors must preserve:
  (a) graph captures the entire decode forward including sampling
  prep handoff, (b) sample readback is async (event-poll), (c)
  next-step embedding feeds from a stable GPU buffer that doesn't
  alias the in-flight copy.
