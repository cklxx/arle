# M5.3b.1 — Metal `sum` lazy + batched backward flush

## Context

M5.3a (`2026-04-20-m5.3a-device-resident-tensor-acceptance.md`) shipped
device-resident `matmul` + `add` on Metal but left every other op CPU-only,
so each non-`matmul` op forced a host readback via its `_forward` fallback.
The bounded-eval test (`metal_single_forward_backward_step_has_bounded_eval_count`)
gated `after_sum == 1` for that exact reason: `sum_last_axis_forward` did
`ensure_host(y)` to compute the reduction on host.

M5.3b.1 ports **only `sum`** (per spec
[`docs/plans/m5.3-device-resident-tensor.md`](../../plans/m5.3-device-resident-tensor.md))
so the canonical training step `y = x @ w; loss = y.sum()` becomes 100%
device-resident through the forward pass — `loss` itself is a Dirty::Device
rank-0 scalar.

## What Worked

### Additive `Backend::sum_all` trait method

A new `fn sum_all(&self, x: &DeviceHandle, shape: &[usize]) -> Result<DeviceHandle>`
declared next to `add` keeps the existing CPU-only `sum_last_axis_forward`
fallback untouched, so callers that still need an axis reduction
(`metal_backend_sum_last_axis_matches_cpu` etc.) keep working with no
half-states. The naming makes the semantic distinction explicit:
`sum_last_axis` reduces along axis `-1`, `sum_all` collapses to a rank-0
scalar.

- `CpuBackend::sum_all` — host sum into a 1-element `DeviceHandle::Cpu`.
- `MetalBackend::sum_all` — `mlx_reshape(x, [size]) -> mlx_sum_axis(0, false)`
  composing into the lazy MLX graph with **zero** `mlx_eval` calls.
- `CudaBackend::sum_all` — eager via host round-trip; `pending-remote`
  CUDA verification per `feedback_bench_every_change.md` convention.

`crates/autograd/src/ops.rs::sum` lost its `store.ensure_host(a)?;`
preflush, and `crates/autograd/src/ops/reduce.rs::sum` was rewritten to
extract scalar metadata (shape, requires_grad) without `Tensor::clone`
(which asserts against `Dirty::Device`), then call `backend.sum_all` and
`store.alloc_device_tensor(vec![], handle)` for a Dirty::Device scalar
output.

### Batched backward flush — `TensorStore::flush_to_host_batch`

The first-pass M5.3b.1 made `sum` lazy but **regressed** d=128 batch=32
Metal step time by ~50% (0.402ms → 0.595ms). Root cause: `tape.backward`'s
`device_ids` loop called `ensure_host` per-id, and each `ensure_host`
crossed the FFI boundary + grabbed `MLX_GUARD` for its own `mlx_eval`.
With M5.3b.1, both `y` and `loss` are Dirty::Device, so what used to be
1 mid-pass `mlx_eval` (sum's host forcing) became 2 sequential per-id evals
in backward (`ensure_host(y)` + `ensure_host(loss)`).

Fix: a new `TensorStore::flush_to_host_batch(&[TensorId])` collects all
Dirty::Device handles, calls `backend.eval(&[handle1, handle2, ...])`
**once**, then per-id `readback`s. MLX evaluates shared upstream nodes
exactly once when given the batched handle slice — `loss` shares all of
`y`'s upstream graph, so the second handle is essentially a tagged read.
`tape::backward` switched to this helper.

Result: steady-state evals/step at this shape go from a regressed 3 back
down to **2** (one batched flush + `mlx_matmul_backward`'s self-eval). The
`metal_single_forward_backward_step_has_bounded_eval_count` test was
tightened from `after_backward <= 3` to `after_backward <= 2` to lock this
in as a regression bound.

### Test coverage

`cargo test --release -p autograd --features metal` — **48/48 green**
(42 CPU-parity + 6 device-handle, including the strict
`after_sum == 0` and `after_backward <= 2` assertions).

`cargo test --release -p autograd` (CPU-only) — **all green**, no semantics
shifted on the CPU path.

### Bench (matched A/B, same session, same Mac)

`crates/autograd/examples/bench_step_matmul.rs` at d=128 batch=32 iters=2000
(longer iters chosen to amortize warmup jitter — short runs of 200 iters
still show single-digit-ms FFI noise dominating the ratio):

| Variant            | CPU ms/step (median) | Metal ms/step (median) | Metal / CPU |
| ------------------ | -------------------: | ---------------------: | :---------: |
| Today's main (M5.3a) | 0.450                | 0.356                  | 1.26×       |
| **M5.3b.1**        | **0.459**            | **0.343**              | **1.34×**   |

Net: ~+6% improvement in Metal/CPU ratio at the smallest acceptance shape,
inside the 5% noise band the spec asked for. Larger shapes are dominated
by GPU compute and unaffected:

| d_model | M5.3b.1 CPU ms | M5.3b.1 Metal ms | Metal / CPU |
| :-----: | -------------: | ---------------: | :---------: |
| 128     | 0.459          | 0.343            | 1.34×       |
| 256     | 2.083          | 0.466            | **4.47×**   |
| 512     | 8.557          | 0.808            | **10.59×**  |

(d=256 / d=512 measured at iters=1000 / iters=500 respectively for
runtime; ratios are stable across iter counts at these shapes.)

The published M5.3a baseline of 1.74× CPU→Metal was on an unloaded Mac
in a different thermal state — today's same-hardware reproduction of
M5.3a is 1.26×, which is the apples-to-apples comparison point.
M5.3b.1's 1.34× clears that within the 5% gate.

## Caveats

1. **`flush_to_host_batch` is wider than M5.3b.1's nominal "one op"
   scope.** The batched flush is a 2-line correctness fix — without it,
   the lazy `sum` is a net regression at small shapes. Per
   `feedback_no_half_states.md`, shipping a regression IS a half-state,
   so the batching landed in the same commit as the lazy port.
2. **Stretch goal `after_backward == 1` still requires lazy
   `Backend::matmul_backward`.** That's M5.3b.x — it would fold the only
   remaining intra-graph `mlx_eval` into the terminal flush.
3. **CUDA `sum_all` typechecks only on Mac host.** Matches the
   `pending-remote` pattern from `2026-04-20-matmul-backward-gpu.md`. CUDA
   path is correct-by-eager-construction (host round-trip), but unrun.
4. **Other CPU-only ops still gate the lazy graph.** `softmax`, `rmsnorm`,
   `gelu`, `silu`, `embedding`, `gather`, `add_broadcast` all still call
   `ensure_host` in `ops.rs`. Each one is its own M5.3b.x deliverable;
   `sum` is the first because it sits at the loss boundary of every
   training tape, where every step gets at least one guaranteed flush.

## Rule

- When making a forward op lazy on a backend with batchable terminal
  flush, audit the tape's flush loop in the same commit. If it walks
  per-id `ensure_host`, it WILL silently regress steady-state perf the
  moment a second Dirty::Device output appears in the relevant tape — the
  regression looks identical to "MLX is slow at small shapes" but is
  actually FFI overhead amplified by the lack of batching.
- Strict `after_backward` bounds in
  `metal_single_forward_backward_step_has_bounded_eval_count` are the
  regression net for the above. The test was tightened from `<= 3` →
  `<= 2` here; tighten further the moment a future change makes it pass
  at the new bound.
- "Within 5% of the published baseline" requires re-measuring main on the
  same hardware in the same session. Published numbers from a different
  thermal state are not comparable — use them as historical context, gate
  on a fresh A/B.
