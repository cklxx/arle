# 2026-04-15 · Metal M0.2a local request-state landing

## Context

After the API/DX tranche and the HTTP sweep validation, the Metal roadmap still
had the same hard blocker: `metal_serve` was serial because the Metal backend
only exposed whole-request `generate_stream`, not resumable per-request state.

The next local tranche for `M0.2a` had to land the **state object** first,
before trying to rewire `metal_serve` onto a live scheduler.

## What Worked

- Added `infer/src/backend/metal/request_state.rs` as the first real
  resumable-request layer for Metal.
- `MetalBackend::create_request_state(...)` now exposes a backend-owned request
  state instead of forcing all execution through whole-request generation.
- Qwen3 request state owns its KV cache / optional KV pool and can:
  - prefill in bounded chunks
  - emit the first sampled token exactly once at terminal prefill
  - continue one-step decode from the committed token
  - clean up deterministically on finish or drop
- Qwen3.5 request state does the same, reusing the existing Rust/MLX step path
  or the compiled C++ step model when available.
- Kept this tranche structural:
  - no `metal_serve` rewiring yet
  - no claim that serving throughput changed
  - scheduler tests stayed green while request-state tests pinned the new
    lifecycle rules
- Verified locally with:
  - `cargo test -p infer --no-default-features --features metal,no-cuda request_state -- --nocapture`
  - `cargo test -p infer --no-default-features --features metal,no-cuda metal::scheduler -- --nocapture`
  - `cargo check -p infer --no-default-features --features metal,no-cuda --bin metal_serve`
  - `git diff --check`

## Rule

For Metal serving milestones, land **request-state ownership before runtime
rewiring**. If the backend cannot prefill/decode/cancel one request as a
resumable state object, swapping `BackendRuntimeHandle` for a scheduler only
changes the wrapper, not the serving architecture.
