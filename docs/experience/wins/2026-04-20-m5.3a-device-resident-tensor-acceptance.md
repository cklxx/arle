# M5.3a device-resident tensor / lazy-eval boundary — acceptance

## Context

M5.3a was commissioned 2026-04-19 after
[`2026-04-18-bench-train-multi-turn.md`](2026-04-18-bench-train-multi-turn.md)
measured Metal at 1.9× slower than CPU on TinyLM training. Root cause: the
pre-M5.3a `Backend::matmul_forward` path uploaded → `mlx_eval` → readback
on every op, so an N-op tape produced N evals — exactly the degenerate
1-op-per-eval pattern that MLX's own
[lazy-eval docs](https://ml-explore.github.io/mlx/build/html/usage/lazy_evaluation.html)
warn against ("tens-to-thousands of ops per evaluation").

Landing commits:

- `17b3599` — M5.3a.1 `DeviceHandle` infra, `Dirty` enum,
  `upload`/`eval`/`readback` on the `Backend` trait, `ensure_host` /
  `ensure_device` on `TensorStore`.
- `7d024d8` — M5.3a.2 lazy `matmul` + `add` composing into the MLX graph
  with no intermediate eval; eval deferred to first `ensure_host` /
  `to_host`.
- `0f3cdf6` (preceding) — `matmul_backward` moved to GPU for Metal so the
  backward matmul no longer round-trips to CPU. See
  [`2026-04-20-matmul-backward-gpu.md`](2026-04-20-matmul-backward-gpu.md).

Spec: [`docs/plans/m5.3-device-resident-tensor.md`](../../plans/m5.3-device-resident-tensor.md).

## What Worked

### Eval-boundary instrumentation + acceptance test

`crates/autograd/src/backend_metal.rs` gained a process-global
`AtomicU64 METAL_EVAL_COUNT` with `bump_eval_count()` wired into every
`mlx_eval` call inside the crate (`MetalBackend::eval`,
`eval_and_readback`, and `mlx_softmax_like`'s direct `mlx_eval`). Exposed
via `pub fn eval_count()` / `reset_eval_count()`.

`crates/autograd/tests/test_device_handle.rs` asserts per-stage bounds
for the reference tape `y = x @ w; loss = y.sum(); backward`:

| Stage              | Expected eval count | Why                                               |
| ------------------ | :-----------------: | ------------------------------------------------- |
| after forward matmul | `== 0`            | matmul composes into MLX lazy graph, no flush.    |
| after `sum`          | `== 1`            | `sum` is CPU-only → `ensure_host(y)` forces flush. |
| after `backward`     | `<= 2`            | `mlx_matmul_backward` does its own eval+readback. |

The stretch goal of strict `<= 1` requires an M5.3b-era lazy
`Backend::matmul_backward` that returns unevaluated `DeviceHandle`s and
defers flush to a terminal tape flush; out of M5.3a scope. Parallel test
pollution of the process-global counter is prevented by a file-local
`METAL_TEST_LOCK: Mutex<()>` that every Metal test in the binary
acquires.

### Acceptance bench (`crates/autograd/examples/bench_step_matmul.rs`)

Isolated training-step micro-bench: `x:[B,d] @ w:[d,d] → sum → backward`,
matching the TinyLM hot path shape. Warmup-then-timed-loop; deterministic
RNG for reproducibility. M5.3a acceptance gate (plan §5): Metal ≥ 1.1× CPU
at d_model=128.

Run: `cargo run --release -p autograd --example bench_step_matmul --features metal -- --backend {cpu,metal} --d <d> --iters 200 --batch 32`.

Results at batch=32 (matches 2026-04-18 TinyLM effective batch of
group=8 × seq=10 ≈ 80):

| d_model | CPU ms/step | Metal ms/step | Metal / CPU |
| :-----: | ----------: | ------------: | :---------: |
| 64      | 0.118       | 0.278         | 0.42× (slower) |
| 128     | 0.509       | 0.293         | **1.74×** (gate passes) |
| 256     | 2.205       | 0.382         | 5.78×       |
| 512     | 9.093       | 0.737         | 12.34×      |

Pre-M5.3a baseline at d=128 was 0.52× (2026-04-18) — M5.3a delivered a
3.3× swing at the acceptance shape. Tail at d=64 still favors CPU
because MLX per-call FFI overhead dominates real work; batch=4 also
stays CPU-favored (d=128 0.42×) — expected, since the whole point of
the lazy graph is to amortize FFI cost across enough ops. Once M5.3b
ports the CPU-only ops (`sum`/`softmax`/`rmsnorm`/`gelu`/`silu`/
`embedding`/`gather`/AdamW) to Backend methods, the per-step eval count
drops to 1 and small-batch Metal should overtake CPU too.

## Caveats

1. **`sum` still forces a readback.** The bounded-eval assertion allows
   `after_sum == 1` precisely because `sum` is CPU-only today and calls
   `ensure_host(y)` on its input. Same story for every op listed in the
   AGENTS.md "When an op forces a host readback" section. M5.3b closes
   this.
2. **Grad accumulation host-side.** `tape.rs::merge_grad` +
   `tensor.rs::accumulate_grad` both `to_host` the incoming grad before
   the final `iter_mut().zip` sum. Tracked as an M5.3b follow-up.
3. **d=64 Metal still slower than CPU.** Expected at the acceptance
   shapes we committed to; the 1.1× gate is d=128+ only. No regression
   — just an honest floor from MLX FFI per-call overhead at trivially
   small tensors.
4. **CUDA path typechecks only (Mac host).** CUDA's matmul_backward-on-
   GPU landed but untested; covered by the
   [`2026-04-20-matmul-backward-gpu.md`](2026-04-20-matmul-backward-gpu.md)
   `pending-remote` status.

## Rule

- When porting a backend op to a lazy / device-resident handle, commit
  forward + backward in the same refactor unit — no half-states
  (`feedback_no_half_states.md`). A lazy forward whose backward
  round-trips to CPU is worse than pure-CPU at small shapes because
  the eval boundary lands inside the backward rather than at a tape
  terminus.
- Maintain the bounded-eval assertion
  (`metal_single_forward_backward_step_has_bounded_eval_count`) as a
  regression signal — if the count climbs above 2, a host round-trip has
  snuck into the middle of the graph, and the trace bisect should start
  at whichever op's `ensure_host(input)` just got called.
- Acceptance benches with MLX backends need realistic batch sizes
  (`feedback_matched_ab_for_small_bench_effects.md`). A d=128 batch=4
  micro-bench would have falsely failed the 1.1× gate; batch=32 tells
  the truth.
