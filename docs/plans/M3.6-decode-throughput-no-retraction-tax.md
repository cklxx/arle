# M3.6 — close the high-conc decode-throughput gap (Nsight-led)

> Discovered 2026-05-07 from codex's evidence-only slot-width sweep
> ([`m6-cuda-vllm-gap-followups.md`](m6-cuda-vllm-gap-followups.md)
> §"Phase 1 verify"). Equal-width vLLM control rejected the
> "admission width alone explains the gap" hypothesis.
>
> **Methodology revision (2026-05-07, user direction)**: lead with
> Nsight Systems profiling, not bench-driven F1/F2/F3 sweeps. The
> industry-standard root-cause path for "scheduler-vs-kernel" gaps in
> LLM inference is `nsys` first, `ncu` second (see §References).
> Bench-and-fix without trace evidence is anti-pattern.

## P0 finding (evidence chain)

Codex's Phase-1 sweep at slots=48 exposed a gap that bench data
alone cannot localize:

| Metric | s48 sweep value |
|---|---:|
| `active` peak (slots holding requests) | **47** |
| `running_batch` peak (slots in scheduler running queue) | **38** |
| `decode` peak (slots advancing one token in launched step) | **38** |
| `kv_util` peak | 95.4% |
| out tok/s | 442 |
| vLLM s14 equal-width control out tok/s | **615** (+39%) |

The gap could plausibly live in any of three layers, and no
service-trace metric distinguishes between them:

| Layer | Hypothesis | Service-metric proxy | Distinguishable from metrics alone? |
|---|---|---|---|
| Scheduler CPU overhead | Mixed-plan retract loop + admission burns Python-equivalent CPU ms per tick | `Idle` plan share, `step_us` distribution | Partial — `Idle` share masks GPU-bound stalls |
| Scheduler policy (page accounting) | `step_mixed_launch` `retract_decode_to_fit(extra_pages>0)` causes decode churn | running_batch=38 vs active=47 gap | Suggestive but not conclusive |
| Decode kernel efficiency | TileLang BF16 attention or GEMV slower than vLLM's Triton at high effective batch | TPOT, ITL p50/p99 | No — same metric value can come from CPU bubble OR slow kernel |

`nsys` resolves all three in one trace by exposing:
- CPU thread time per scheduler function (catches WukLab-style
  "scheduling overhead dominates" pattern)
- GPU bubble periods between kernel launches (catches CPU-bound
  scheduling stalls)
- Per-kernel duration grouped by name (catches per-kernel inefficiency)
- CUDA Graph replay validation (catches static-shape regressions)

## Plan

### Phase 0 — install + dry-run

```bash
sudo pacman -S nsight-systems nsight-compute   # cachyos packages
nsys --version && ncu --version                # confirm
```

Already-installed CUPTI (`/opt/cuda/extras/CUPTI`) plus the toolchain
checked above.

Dry-run smoke test on a 5-token completion to confirm tooling works
end-to-end before the real high-conc capture.

### Phase 1 — `nsys` capture at the s48 high-conc shape

Match codex's Phase-1 conditions exactly so the trace and bench data
overlap:

```bash
RUST_LOG=info NVCC_CCBIN=/usr/bin/g++-14 \
INFER_TILELANG_PYTHON=/home/ckl/projects/arle/.venv/bin/python \
TORCH_CUDA_ARCH_LIST=8.9 \
nsys profile \
  --trace=cuda,nvtx,osrt,cublas,cudnn \
  --cuda-graph-trace=node \
  --capture-range=cudaProfilerApi \
  --capture-range-end=stop \
  --output=/tmp/arle-s48-highconc \
  --force-overwrite=true \
  target/release/infer \
    --model-path /home/ckl/projects/arle/infer/models/Qwen3-4B \
    --port 8000 \
    --max-seq-len 2048 \
    --num-slots 48
```

Drive with the same guidellm shape from
[`m6-highconc-phase1-arle-2048-s48`](../experience/wins/) (1024 in /
256 out, c=64, 45 s, 5 s warmup). Capture 30 s steady-state via
`cudaProfilerStart` / `cudaProfilerStop` calls — added to ARLE main
behind a new `--nvtx-capture-window-secs N` flag if absent.

Required NVTX ranges (added if missing):
- `step_total` — full scheduler tick
- `step_admission` — `try_admit_waiting`
- `step_plan` — `select_step_plan`
- `step_mixed_launch_retract` — the retract loop in `decode.rs:387-413`
- `step_decode_kernel_launch` — `launch_decode_batch_from_tokens`
- `step_prefill_kernel_launch` — `step_prefill_batch`
- `step_dispatch_emits` — `dispatch_decode_emits`

These are cheap (~ns per range) and let `nsys stats` produce a
clean breakdown. Adding the NVTX scaffolding is the only code
change in Phase 1.

### Phase 2 — vLLM s14 equal-width control trace

Same `nsys` flags, vLLM with the canonical command from vLLM's
profiling docs:

```bash
VLLM_WORKER_MULTIPROC_METHOD=spawn \
nsys profile \
  --trace-fork-before-exec=true \
  --cuda-graph-trace=node \
  --output=/tmp/vllm-s14-highconc \
  vllm serve <args from codex's m6-highconc-phase1-vllm-s14>
```

This is the comparison baseline: where does vLLM spend its time at
the same effective decode width?

### Phase 3 — analyze, then fix

Open both traces in Nsight Systems UI. Categorise:

| Trace finding | Implied fix tier |
|---|---|
| ARLE has GPU bubbles > 50 µs between decode kernel launches that vLLM does not | F1 (scheduler CPU overhead — instrument what blocks the launch) |
| Mixed-step retract NVTX range is wide and recurrent | F2 (page accounting policy — see candidate fixes below) |
| ARLE decode kernel duration > vLLM decode kernel duration at same batch | M3.5 / M_b kernel-level work (separate plan; do not confuse with M3.6) |
| ARLE `step_admission` >> vLLM equivalent | scheduler structure (loop collapse — already covered by M3.5) |

Only after the trace identifies the layer, pick the matching fix
candidate from §Candidate fixes below. Do not stack fixes
speculatively.

### Phase 4 — ncu kernel deep-dive (only if Phase 3 points to kernel layer)

If and only if Phase 3 trace shows decode-kernel duration is the
dominant cost, profile the hot kernel with `ncu --set full` on a
single decode launch. Do not run `ncu` on the full server — it
serializes every kernel for full counter capture and is unusable
for end-to-end runs. Hand off to M_b / M3.5 with the ncu report.

## Candidate fixes (parked until trace evidence selects one)

Each is a hypothesis that a specific trace finding would license. None
should land before its evidence does.

### F1 — request-count cap on concurrent prefill

Set `prefill_max_requests` default = `min(2, num_slots / 16)` instead
of unbounded.

- **Trace evidence required**: NVTX `step_admission` shows multiple
  prefill requests admitted per tick at high kv_util, AND
  `step_mixed_launch_retract` is wide.
- **Mechanism**: `PrefillBudget::from_scheduler_for_decode_slots`
  already wires `prefill_max_requests` into `StepTokenBudget`
  (`execution.rs:144-153`); just changing the default is sufficient.
- **Note**: codex's currently-running sweep at s=48 with
  `--prefill-max-requests=1|2|4` IS this validation, but as a
  bench-only check. Once the trace lands, codex's sweep results map
  cleanly to the fix decision.

### F2 — bias `select_mixed_launch_prefill_candidates` against decode-retraction

When `kv_util > 0.85`, refuse to admit a prefill candidate whose
`additional_pages_needed_for_slot` would force retraction.

- **Trace evidence required**: `step_mixed_launch_retract` NVTX
  range fires every tick at high kv_util; tracing GPU bubble inside
  the retract loop.
- **Mechanism**: in `step_mixed_launch` (`decode.rs:374`) before
  the retract loop, filter out candidates whose `extra_pages` would
  push pool past 100% — only when `kv_util > threshold`.
- **Risk**: starve prefill if kv_util stays above threshold
  permanently. Mitigation: max prefill defer count per slot
  (e.g. 4 ticks); after that fall back to old retract behavior.

### F3 — split prefill page budget from decode pool entirely

vLLM-style: reserve a fixed fraction (e.g. 5-10%) of the KV pool as
"prefill scratch" that decode never sees. Mixed steps allocate
prefill into the scratch pool; decode never gives up pages.

- **Trace evidence required**: even with F1+F2, GPU bubble around
  page-allocation paths is dominant; `additional_pages_needed_for_slot`
  shows up in NVTX as a hot range across many ticks.
- **Mechanism**: `PageBudget::from_scheduler` becomes two budgets:
  `decode_budget` (95%) and `prefill_scratch_budget` (5%).
- **Cost**: ~150 LOC; touches `PageBudget`, `PrefillBudget`,
  page-allocator hooks. New CLI: `--prefill-scratch-fraction 0.05`.
- **Risk**: 5% pool reserved means slightly fewer concurrent decode
  slots fit. At s48, ~10 tokens per slot of decode capacity loss.

## Bench gate

After each landed fix:

1. `scripts/bench_guidellm.sh m3.6-<f1|f2|f3>-<label>` — high-conc
   shape (1024 in / 256 out, c=64), `--num-slots 48` and 64.
2. Compare against `m6-highconc-phase1-arle-2048-s48` (442 out tok/s)
   and `m6-highconc-phase1-vllm-s14` (615 out tok/s).
3. Wins entry MUST cite both baselines with Δ% rows AND link the
   nsys trace artifact path.
4. Regression on TTFT p99 > +20%, e2e correctness, greedy_consistency,
   or M4.5 drain → revert; trace must explain the regression.

## Acceptance (whole milestone)

- ARLE high-conc out tok/s at s48 ≥ 0.95 × vLLM s14 throughput
  (≥ 584 out tok/s).
- TTFT p99 at s48 ≤ baseline + 20%.
- All in-tree correctness tests still green.
- Two nsys captures committed as artifacts (ARLE before-fix, ARLE
  after-fix); vLLM control trace cited in wins entry.

## Out of scope

- Decode-kernel-level optimizations (TileLang attention, GEMV
  vectorization). M3.6 is scheduler policy + observability only.
  Kernel work routes through M3.5 / M_b after Phase 4 ncu report.
- Spec-decode interactions. `step_mixed_launch` runs only on the
  non-spec path; spec paths have their own `retract_decode_to_fit`
  callers (`spec_path.rs:95, 343`) and stay untouched.
- Multi-GPU / tensor-parallel scheduling.
