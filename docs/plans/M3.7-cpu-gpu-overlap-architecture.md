# M3.7 — CPU/GPU overlap architecture (post-F4-Small)

> Sequel to [`M3.6-F4-eliminate-per-step-sync.md`](M3.6-F4-eliminate-per-step-sync.md).
> F4-Small landed (`2a534c4`) and produced +82.5% throughput
> (`8f83c80`: ARLE 843 vs vLLM s14 647) by removing per-step sample
> sync. This plan designs the **next level** of CPU/GPU overlap.
>
> Goal: characterize what concurrency F4-Small actually achieved
> and design the residual reductions in three tiers (B → A → C),
> with a clear recommendation on what to land next.

## Current state (post-F4-Small)

### What F4-Small achieves

`sample_batch_greedy_launch` + `sample_batch_greedy_readback`
(decode.rs:296 + 690) form a 2-phase split:
- Launch fires argmax kernel + D2D `argmax_out → next_decode_meta_gpu`
  on the compute stream.
- Readback uses `copy_stream` + CUDA event poll (no compute-stream
  blocking).

So per decode tick:
- CPU: ~6 ms (admission + plan + decode_meta H2D + graph launch)
- GPU on compute stream: ~76 ms (graph replay = 36 layers + sampling)
- GPU on copy stream: ~few µs (token D2H, runs in parallel with next tick's GPU work)
- CPU blocks: only if it queues more work than the GPU can consume

This is **Option B-light** (CPU/GPU async via lazy readback). It
unblocks CPU on the readback side. But there's still serialization
on the **launch** side.

### What still serializes

Re-survey of `ctx.sync()` calls in the hot path
(`grep -rn ctx.sync infer/src/{model,scheduler}/cuda/`):

| File:line | Call | Hot? | Status |
|---|---|---|---|
| `model/qwen3/forward.rs:510` | `sample_batch_greedy` (legacy unified) | yes if scheduler hits it | F4-Small only fixed split path; legacy is fallback |
| `model/qwen3/forward.rs:156` | `forward_decode_one_logits` | spec-decode draft only | not high-conc hot path |
| `model/qwen3/forward.rs:479` | `sample_batch_with_penalties` | non-greedy hot path | uncovered by F4-Small |
| `model/qwen3/prefill.rs:121, 380, 391` | prefill chunk submission | yes (chunked prefill) | uncovered |
| `model/qwen3/weights.rs:*` | weight load | boot only | not hot |

So F4-Small **fixed only 1 of ~5 hot sync sites**. The headline
+82.5% gain came from that one site because it dominated. But:
- **non-greedy decode** still per-step syncs
- **chunked prefill** still per-step syncs
- ARLE's high-conc bench is greedy-only at the s48 shape; at
  realistic prod traffic with sampled outputs, the gain shrinks.

### What stream concurrency exists

`crates/cuda-kernels/src/tensor.rs::DeviceContext`:
- `stream` (compute): all kernels + CUDA Graph
- `copy_stream`: async H2D/D2H
- `copy_waits_for_compute()` event-orders copy after compute

**Critical comment in tensor.rs**: "Disable multi-stream event
tracking ... We use a single compute stream, so no cross-stream
synchronization is needed. This avoids `stream.wait(event)` calls
that break CUDA Graph capture."

→ The single-compute-stream is **deliberate** to keep CUDA graphs
working. Adding stream B for decode kernels would break the graph.

## The three-tier overlap menu

### Tier B — Deeper async CPU prep (low risk, build on F4-Small)

Already partially landed by F4-Small. Remaining work:

#### B.1 — Cover the other sync sites

Apply the F4-Small pattern to the remaining hot sync calls:

| Site | Pattern | LOC est. |
|---|---|---|
| `sample_batch_with_penalties` | Same split (launch + async readback) as greedy | ~80 |
| `prefill.rs:380, 391` (per-chunk syncs) | Move to async readback for chunk completion checking | ~50 |
| Legacy `sample_batch_greedy` (forward.rs:510) | Either delete (scheduler uses split path) OR mirror the split | ~30 |

After B.1: all hot decode + prefill sync sites use async pattern.
Expected: another +5-15% throughput at workloads that hit those
paths (non-greedy, prefill-heavy).

#### B.2 — CPU prep for tick N+1 starts before tick N completes

Currently the scheduler tick is:
```
loop {
    plan = select_step_plan(...)        // CPU, ~50µs
    launch_decode_batch(plan)            // CPU, ~1ms (graph replay)
    poll_emits()                         // CPU, ~few µs
    /* GPU runs in background */
    /* implicit wait at next loop iteration's CPU work */
}
```

Even without explicit sync, the CPU runs serially through these.
The "implicit wait" is just CPU naturally going slower than GPU at
times.

B.2 adds **explicit prefetch**: while tick N's GPU work is in flight,
CPU runs tick N+1's `select_step_plan` and pre-stages its
`decode_meta` H2D. The actual `launch_decode_batch` for N+1 still
waits for N to finish via stream queueing (no sync needed; CUDA
serializes within a stream).

- **LOC est.**: ~150 (refactor scheduler loop into prep+launch
  phases + double buffer for decode_meta)
- **Expected gain**: 5-20% on workloads where CPU prep > GPU work
  (small batches, simple kernels). At high-conc with batch ~38,
  gain is small because GPU work already dominates.

**Tier B total**: ~310 LOC, 1-2 days. Best if F4-Small generalizes
fully (all sync sites covered) and CPU prep is hidden under GPU.

### Tier A — Two-stream pipeline (medium risk, breaks current graph capture)

Run **decode kernels on alternating streams A/B** to overlap GPU
execution itself. Tick N runs on stream A; tick N+1 runs on stream B.
Cross-stream ordering via CUDA events:
- Tick N's KV cache writes complete before tick N+1's attention
  reads them
- Tick N's sampled token reaches tick N+1's embedding input via
  event-wait

#### Why CUDA Graph is the obstacle

cudarc's note: `stream.wait(event)` breaks CUDA Graph capture.
TileLang and our hand-CUDA decode all rely on the graph for
launch overhead amortization (graph capture takes 36 layers ×
~5 kernels = 180 launches and replays them as one driver call).

Two ways out:
1. **Two graphs, one per stream**: capture twice. Stream A graph
   contains launch sequence with explicit event-wait inserted at
   the embedding input read. Stream B graph waits on stream A's
   "sample token written" event.
2. **Drop graphs, embrace 5× per-step launch overhead**: trade
   for 1.5-2× pipeline parallelism. Probably net negative on
   small models like Qwen3-4B; could win on bigger models where
   per-kernel time is much larger than launch overhead.

#### Design sketch

```rust
struct PipelineDecode {
    stream_a: CudaStream,
    stream_b: CudaStream,
    graph_a: CudaGraph,
    graph_b: CudaGraph,
    sampled_event_a: CudaEvent,  // recorded after stream_a's sample completes
    sampled_event_b: CudaEvent,
    parity: u8,  // 0 → next launch goes to A; 1 → B
}

impl PipelineDecode {
    fn step(&mut self, ...) {
        let (this_stream, this_graph, this_event, prev_event) = if self.parity == 0 {
            (&self.stream_a, &self.graph_a, &self.sampled_event_a, &self.sampled_event_b)
        } else {
            (&self.stream_b, &self.graph_b, &self.sampled_event_b, &self.sampled_event_a)
        };
        // Wait for prev tick's sample to be ready (event-wait, no CPU sync)
        this_stream.wait(prev_event);
        // Launch this tick's graph (replay)
        this_graph.launch(this_stream);
        // Record this tick's sample event for next tick
        this_event.record(this_stream);
        // Flip parity for next call
        self.parity ^= 1;
    }
}
```

- **LOC est.**: ~400 (PipelineDecode + dual graph capture + event
  management + scheduler integration)
- **Expected gain**: theoretical 2× throughput at high-conc
  (overlapping ticks). In practice 1.3-1.6× because some kernel
  time still serializes on the same SM (4080S has limited SM
  count = 80, so two ticks competing for SMs reduces per-tick
  speed).
- **Risk**: KV cache write→read ordering across streams is the
  delicate part. Wrong event placement = silent corruption (one
  tick reads stale KV from before previous tick's update).
  E2E correctness test must be very strong.

### Tier C — Persistent decode kernel (high risk, radical)

Replace per-tick kernel launch with a **long-running decode kernel**
that loops internally on a GPU-side work queue. CPU writes new
batch metadata to a GPU ring buffer; the kernel wakes up, processes,
writes results, loops.

- Eliminates ALL per-tick launch overhead (currently ~1 ms)
- GPU never idles on launch latency
- **Reference**: NVIDIA Megatron's persistent kernels, vLLM v0.6's
  persistent batch executor (experimental)
- **Risk**: 4080S consumer card, persistent kernels can starve
  other GPU work; SMs locked into the loop. For dedicated
  inference boxes ok; for shared GPU dev boxes painful.
- **LOC est.**: ~800+ (new kernel design + queue protocol +
  scheduler rewrite)

Defer C until B+A together leave a known gap that only persistent
kernels can close. At our current 843 out tok/s headline, C is
overkill.

## Recommended sequencing

```
DONE  ── F4-Small         (per-step sample sync removed) +82.5%
NEXT  ── B.1              (cover remaining sync sites)    +5-15%
THEN  ── B.2              (explicit CPU prefetch tick N+1) +5-20%
LATER ── A                (two-stream pipeline)            +30-60% (if needed)
NEVER ── C                (persistent kernel)              defer until A insufficient
```

Reasoning: each tier is independently testable, B-tier is
incremental on F4-Small (low risk, no architecture change), A-tier
is the big architectural lift (worth it when B-tier saturates).

## Acceptance

### B.1
- All hot sync sites in `model/qwen3/{forward,prefill}.rs` use
  async pattern.
- `cargo test --features cuda --test e2e` and `--test greedy_consistency` pass.
- Bench delta vs F4-Small baseline (843 out tok/s):
  - high-conc greedy: ≥ +0% (no regression)
  - non-greedy or prefill-heavy: ≥ +5%
- nsys trace: `cuStreamSynchronize` ≤ 10% of CPU API
  (down from ~25% post-F4-Small estimated).

### B.2
- No regression on greedy_consistency.
- Bench at c=4/c=8 small-batch shapes: ≥ +5% throughput.
- Bench at c=64 high-conc: no regression (GPU-dominated workload
  doesn't benefit but must not lose).

### A
- `cargo test --features cuda --test e2e` passes (KV cache ordering
  correctness).
- `cargo test --features cuda --test greedy_consistency` passes
  (cross-stream sampled-token handoff correctness).
- Bench: ≥ +30% throughput at high-conc vs F4-Small + B.

## Risks + retreat

- **R1 — B.1 breaks non-greedy correctness**: penalty sampling has
  sequential dependencies (prev-token frequency, repetition count)
  that may not survive 1-tick-deferred readback. Mitigation: add a
  `--decode-pipeline-mode` flag (sync/async) and gate non-greedy
  to sync until proven.
- **R2 — B.2 prefetch races with retraction**: prefetched tick N+1
  state assumes the running_batch shape from tick N+1's plan. If
  tick N+1 retracts (rare but possible), the prefetch is wasted.
  Mitigation: prefetch-and-discard is correct (just throws away
  CPU work); no correctness risk.
- **R3 — A wrong event placement**: corrupted KV reads. Mitigation:
  add a CUDA-event-ordering invariant test that runs a synthetic
  pattern where tick N+1's expected output depends on tick N's
  exact write — assert byte-exact match.
- **R4 — A reduces single-tick latency**: pipeline parallelism
  helps throughput but each individual tick may run slower (SM
  contention). For latency-sensitive workloads (low-conc, c=1-4),
  A could be net negative. Acceptance includes c=4 latency check.

## Out of scope

- F4-Big (in-graph sampling): superseded — F4-Small already
  exceeded plan's parity target. F4-Big was a bigger-hammer
  alternative; B.1 is the smaller-hammer evolution.
- Tier C (persistent kernel): only if A + B leave a known gap.
- Multi-GPU pipeline parallelism (different model layers on
  different GPUs): out of scope; single-GPU only.
- Metal backend: M3.7 is CUDA-only; Metal has its own queue
  semantics and benefits less from these patterns.
