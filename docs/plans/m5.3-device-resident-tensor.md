# M5.3a — Device-resident tensor + lazy-eval boundary

**Status**: draft · 2026-04-19
**Owner**: ckl
**Parent**: [`rust-agent-rl-single-node.md`](rust-agent-rl-single-node.md) §7.2 M5.3
**Blocks**: M5.3b (batch-land Metal ops). M5.4 Mac LoRA validation was closed on 2026-04-21; see `docs/experience/wins/2026-04-21-qwen35-metal-lora-validation.md`.

---

## 1. Problem

`crates/autograd/src/backend_metal.rs::matmul_forward` uploads host `&[f32]`
into `mlx_array_from_data`, calls `mlx_matmul`, forces `mlx_eval`, copies
the result back to `Vec<f32>`, frees all three arrays — per call.

On TinyLM-scale shapes this
makes Metal **1.9× slower than CPU**, breaking even only at d_model ≥ 256
(1.4×). The bottleneck is FFI + 1-op graph build + readback, not the GPU.

Upstream confirms this is degenerate usage. MLX's own
[lazy-evaluation guide](https://ml-explore.github.io/mlx/build/html/usage/lazy_evaluation.html):

> "There is some fixed overhead with each graph evaluation … we recommend
> tens-to-thousands of ops per evaluation."

Our current shape = 1 op per eval. Fix = batch ops into one eval boundary.

---

## 2. Scope

**In scope**
- `crates/autograd/src/tensor.rs` — extend `GpuTensor` with optional device
  handle.
- `crates/autograd/src/backend.rs` + `backend_metal.rs` — add device-to-device
  op methods that operate on handles, not `&[f32]`.
- Eval boundary — one explicit `flush`/`eval` point per forward+backward
  pass, called from `Tape::backward()` (or `optimizer_step()`).
- Ops touched by M5.3a: **matmul only**. No new ops yet.

**Out of scope (→ M5.3b)**
- Porting add / mul_scalar / sum / log_softmax / gather / AdamW to Metal.
- Mixed-device tapes (CUDA + Metal in one tape).
- Multi-stream / async eval.
- Fused kernels / custom Metal shaders.

**Deliberately unchanged**
- Tape structure (`TapeEntry`, `BackwardOp`, `SavedContext`) — M5.3a is
  a backend-side refactor; the tape/op API stays stable.
- CPU backend — host-`Vec<f32>` path is the reference; grad_check still
  runs against it.
- CUDA backend — keep the existing upload/compute/download path;
  device-resident CUDA is a separate future task.

## 3. Architecture

### 3.1 Tensor representation

```rust
// crates/autograd/src/tensor.rs
pub struct GpuTensor {
    pub shape: Vec<usize>,
    pub strides: Vec<usize>,
    pub size: usize,
    pub requires_grad: bool,
    pub grad: Option<TensorId>,

    // NEW: one of these is authoritative; the other may lag.
    host: Option<Vec<f32>>,         // canonical on CPU backend; cache on GPU
    device: Option<DeviceHandle>,   // populated on GPU backends after upload
    dirty: Dirty,                   // which side has the latest bytes
}

enum Dirty { Host, Device, Both }

pub struct DeviceHandle {
    backend: Device,         // Metal | Cuda
    // Opaque raw pointer + destructor; Metal wraps mlx_array.
    // One handle = one live array; cloning requires explicit op.
    ...
}
```

Invariants:

- `Dirty::Host` → `host = Some(_)`, `device` may be `None` or stale.
- `Dirty::Device` → `device = Some(_)`, `host` may be `None` or stale.
- `Dirty::Both` → both are populated and agree (after a flush+readback, or
  after `from_slice` on CPU-only tape).
- On drop, device handle runs its destructor (`mlx_array_free` for Metal).

### 3.2 Eval boundary

One explicit flush per tape step:

```
forward:    build graph via device-to-device ops        (no eval)
backward:   append grad ops to the same graph           (no eval)
flush:      mlx_eval(<all outputs we need on host>)     (single eval)
readback:   lazily, only when to_host() is called       (marks Dirty::Both)
```

Flush is driven from `Tape::backward()` end-of-call. For training we only
need host copies of: (a) loss scalar, (b) gradients going into
`optimizer_step`. Intermediate activations stay device-resident and get
freed when the tape drops them.

### 3.3 Backend trait shape

Replace per-call `&[f32]` API with handle-taking methods:

```rust
pub trait Backend: Send + Sync + Debug {
    fn device(&self) -> Device;

    // Upload: only when tensor originates on host (weights, inputs).
    fn upload(&self, host: &[f32], shape: &[usize]) -> Result<DeviceHandle>;

    // Device-to-device ops. NO eval inside. Return lazy handle.
    fn matmul(&self, a: &DeviceHandle, b: &DeviceHandle) -> Result<DeviceHandle>;

    // Eval boundary. Evaluates all passed handles in one MLX call.
    fn eval(&self, handles: &[&DeviceHandle]) -> Result<()>;

    // Readback: only when host needs the bytes. Implies eval.
    fn readback(&self, handle: &DeviceHandle) -> Result<Vec<f32>>;
}
```

CPU backend: `upload` and `readback` are no-ops on a wrapper that holds
`Vec<f32>` directly; `eval` is a no-op.

Metal backend: `upload` = `mlx_array_from_data`; `matmul` = `mlx_matmul`
(returns handle to unevaluated graph node); `eval` = one `mlx_eval` over
all terminal handles; `readback` = `mlx_array_data_float32` + copy.

### 3.4 CPU backend under the new API

To keep one tape code path, CPU wraps `Vec<f32>` as a `DeviceHandle`
variant. `upload`/`readback` are zero-copy moves; `eval` is a no-op.
This keeps `grad_check` against CPU as the reference without branching in
the tape.

## 4. Phased landing

### M5.3a.1 — Handle type + upload/readback (small)

- Add `DeviceHandle` enum (CPU/Metal/Cuda variants; Cuda unimpl! for now).
- Add `upload` + `readback` + no-op `eval` to the trait.
- CPU backend: trivial wrapping.
- Metal backend: port existing upload/readback into the new methods; leave
  the old `matmul_forward` in place during this step.
- `GpuTensor` carries both `host` and `device` fields but the tape still
  uses `host` only — **no behavior change, just infrastructure**.
- Acceptance: `cargo test -p autograd` (CPU + Metal) unchanged results.

### M5.3a.2 — Device-resident matmul + eval boundary

- `Backend::matmul(&DeviceHandle, &DeviceHandle) -> DeviceHandle` on Metal.
- Tape `matmul` op: if both inputs have `device`, produce output as
  `Dirty::Device`; otherwise upload then call.
- `Tape::backward()` ends with a single `backend.eval(terminal_outputs)`;
  `readback` happens on `to_host` calls from the training loop / optimizer.
- Delete `matmul_forward` from the trait; migrate callers.
- Acceptance:
  1. `cargo test -p autograd --features metal` green (grad_check vs CPU).
  2. `bench_train_multi_turn` at TinyLM shape: Metal ≥ CPU (was 1.9× slower).
  3. `mlx_eval` calls per step = 1 (measure via a counter in the backend).

### M5.3a.3 — Cleanup + docs

- Remove `matmul_forward` trait leftovers if any.
- Update `crates/autograd/AGENTS.md` with the new handle contract + eval
  boundary rules.
- Cross-link this doc from `rust-agent-rl-single-node.md` §7.2 row M5.3.

## 6. Risks + mitigations

| Risk | Mitigation |
|---|---|
| MLX handle lifetime across tape steps — eval-then-free vs eval-then-reuse | Reference-count in `DeviceHandle::Drop`; tape drops intermediate handles at step end |
| CPU backend paying upload/readback overhead on unified API | `DeviceHandle::Cpu(Vec<f32>)` + specialized `Backend::eval` no-op; zero-copy |
| Grad accumulation in `accumulate_grad` currently does host-side `iter_mut().zip` — breaks if grad stays device-resident | Route grad accumulation through a device `add` op *in M5.3b*; until then, force readback in `accumulate_grad` (slow path, but correct) |
| Thread-safety: `MLX_GUARD` mutex in current Metal backend | Keep the mutex; single-threaded training tape means contention = 0 |
| CUDA backend untouched but must still compile | Keep `CudaBackend::matmul` calling existing path; new trait methods return `unimplemented!()` for CUDA until a parallel CUDA device-resident task spins up |

## 7. Open questions (resolve during M5.3a.1)

- **DeviceHandle Clone semantics**: should `GpuTensor::clone_tensor` do a
  device-to-device copy or share the handle? Proposal: explicit device
  `copy` op; `Clone` on `GpuTensor` only clones metadata + forces readback.
- **Mixed-backend tapes**: forbidden at M5.3a; `TensorStore::set_backend`
  must reject if any live tensor has a device handle from a different
  backend. Panic-on-misuse is fine (internal API).
- **Non-matmul ops on tape**: while they remain CPU-only (pre-M5.3b), every
  CPU op forces a readback of its device inputs. Log these in debug mode
  so M5.3b prioritizes the hottest ones.

## 8. Non-goals (won't do in M5.3a)

- Chunked / streaming eval (one eval per N steps) — premature until we
  have profile data post-M5.3b.
- Per-op FLOP counters / tensorboard-style tracing — use existing bench
  scaffolding.
- Metal kernels beyond what MLX exposes — custom `.metal` stays out of
  `autograd` per `crates/mlx-sys/AGENTS.md`.
